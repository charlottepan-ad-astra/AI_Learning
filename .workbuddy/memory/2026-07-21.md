## Sherpa AI — 聊天框 System Prompt 二次优化

**背景：** 用户在聊天框里直接给出了 5 模块的 Sherpa Prompt，要求"在原基础上做二次优化"，并强调"保持语言一致性"（之前提过）。

**做了什么：**
- 在 `server.js/server.js` 的 `/api/chat` 路由里，把 5 模块的原始 Prompt 升级到 6 模块的 v2。
- 原始 5 模块全部保留（结构不破），新增 Module 6「Session Awareness & Edge Cases」处理：跑题/发泄、重复问、半路想放弃、平台工具（学习地图/错题本/测验）的自然引导。
- DDA 表格化：原版 3 个 IF 分支升级为 4 行信号表（增加 Deepening Mode 给"答对 + 要深度"的用户）。
- 加 Response Format 模板：Diagnose / Substance / Why / Next Step 四段式 + 150-400 词长度预算。
- 加 Spaced Re-trigger 规则到 Module 5，复访老话题时主动回扣上下文。
- 加 Initialization 第 3 题（10/30/60+ 分钟），决定节奏。
- Language Consistency 规则加"技术词可保留原文"的弹性，避免来回切语。
- 加 Session Context 段，让模型把当前会话历史当记忆来用。
- 把优化后的 Prompt 单独存到 `prompts/sherpa-system-prompt.md`，带 v1→v2 对比表，方便后续维护。

**关键决策：**
- 系统层 Prompt 保持英文（工程/指令层），用户消息语言由 Language Consistency 规则控制。
- DDA 信号表用 Markdown 表格，R1 这类推理模型解析更稳。
- 平台工具提示"每个 session 最多一次"，避免变成广告。

**验证：**
- `node -c server.js/server.js` 语法检查通过。

## Bug 修复：切换账号后旧数据残留

**用户反馈：** 创建新账号后，聊天框内容 + learning map 依然保留旧账号的对话历史。

**根因（两层叠加）：**
1. `clearAuthState()`（line 1515）只清 `CURRENT_USER_ID` / `CURRENT_PLAN_ID` + 切 UI 显隐，不动任何业务态（state.chatHistory / learningGoal / suggestedReplies / learningNodes / hasLearningGoal / mistakes / resources / currentQuestion）。
2. 4 个 fetch 函数（`fetchChatHistory` / `fetchLearningMap` / `fetchMistakes` / `fetchResources`）是"覆盖式" — 只有在 `data && data.length` 时才赋值；新账号没数据时，旧值原地不动。`hasLearningGoal` 这个 flag 也是同理。

**修复（在 index.html）：**
- 新增 `resetUserState()` 函数（line 1145）：把所有 in-memory 业务态恢复成初始值，模拟"全新打开页面"的样子；同时给 `DEMO_SESSION_ID` 重新生成 UUID。
- `onAuthSuccess()` 第一行调一次 `resetUserState()`（双保险，兼容刷新直接登录场景）。
- `clearAuthState()` 在置空 CURRENT_USER_ID 之后调一次 `resetUserState()`。
- 4 个 fetch 函数都加了 else 分支：拿不到数据时显式把对应数组/对象清空（防御性，避免未来加 refetch / 热重载时再踩坑）。
- `DEMO_SESSION_ID` 从 `const` 改为 `let`，允许重新生成。

**验证：**
- `node .workbuddy/scratch/parse_check.js` 静态解析通过（412/412 brace 对齐）。
- 后端 3000 端口仍在跑（PID 29244）。
- 浏览器端没条件做 e2e，等用户回前端手测。

**经验（记下来）：**
- 前端"覆盖式 fetch"是常见的账号切换数据残留坑。模式：用单一 `resetState()` + 在 auth 边界（login/logout）调用 + fetch 里 else 显式清空 = 三重保险。
- 类似的"全量重置"逻辑未来加新业务态时，记得同步更新 `resetUserState()` 里的列表。
- 用户的复测建议：登出账号 A → 登录/注册账号 B → 看 4 个页面（workspace 聊天、learning map、mistake book、resources）是否都回到"全新用户"的空态。

## 产品逻辑修复：chat-first 学习计划（用户报"新账号 quiz/learning map 自己冒出来"）

**用户反馈：** 新账号进入后，AI Generated Quiz 和 learning map 在几秒后自动出现（鼠标动一下就看到了），但用户此时还没和 AI 聊过任何东西。

**产品逻辑（用户重新明确的）：**
- AI 应该先和用户聊天，了解用户兴趣
- 然后再生成专属学习题目（题目+知识点）、动态学习地图
- 配错题总结、相关学习资料

**根因：**
- 老代码 `getOrCreateActivePlan()` 在新账号注册时**强制塞 4 个 demo 概念**（python_basics / statistics / ml_fundamentals / neural_networks）到 plan 里。
- 这导致 `fetchLearningMap` 一查就返回 4 个节点 → `hasLearningGoal = true` + `learningNodes` 填满 → learning map 渲染。
- `loadNextQuestion` → `pickActiveConcept` 返回第一个概念 → 随机抽题 → quiz 区渲染。
- 整个过程在网络请求回来后（约 1-2 秒）就"啪"一下完成了，违反 chat-first 产品逻辑。

**修复（在 index.html）：**
1. **`getOrCreateActivePlan` 不再 attachDemoConceptsToPlan**，plan 创建出来是空的，subject/goal 都设为 null（等用户聊完再填）。
2. **新增 `topicToConceptKeys(topic, goalText)`**：把 AI 推断的 topic 映射到 DB 里的 concept_key。覆盖 Python / Statistics / ML / SQL / JS 五个 case，兜底返回 `ml_fundamentals`。
3. **新增 `attachConceptsForTopic(planId, topic, goalText)`**：按 topic 查 DB 里的 concept，幂等地插入 `learning_plan_concepts`（跳过已挂的，算下一个 sequence_order）。
4. **新增 `persistInferredGoal(inferred)`**：在用户聊完天后，异步地把 `learning_plans.subject / .goal` 更新 + 挂载概念 + 重新拉 learning map。这是 fire-and-forget，UI 已经立刻给了反馈。
5. **新增 `fetchPlanMeta()`**：`loadAllFromDatabase` 并行调用，从 `learning_plans` 表读 subject/goal 灌到 `state.learningGoal`。这样用户聊过后再刷新页面，依然能看到自己的目标（goal 为空时保持 "Not set yet"）。
6. **`applyCoachGoalAndQuestion` 增加 `persistInferredGoal(inferred)` 调用**：聊天后立刻把目标写回 DB。
7. **`loadNextQuestion` 加防御**：`state.learningGoal === "Not set yet"` 时直接 return null，不让 DB 的旧数据或竞态把 quiz 撑出来。
8. **`attachDemoConceptsToPlan`** 保留但标记为废弃，方便回滚或测试。

**验证：**
- `parse_check.js` 静态解析通过（430/430 brace 对齐）。
- 关键函数都在位：topicToConceptKeys (1 def + 1 call), attachConceptsForTopic (1 def + 1 call), persistInferredGoal (1 def + 1 call), fetchPlanMeta (1 def + 1 call), applyCoachGoalAndQuestion 现在调 persistInferredGoal。
- 后端 3000 端口仍在跑（PID 29244）。

**用户测试路径：**
1. 新注册一个账号 → 应直接看到 workspace 聊天（左）+ "No quiz yet 📚" 占位（右）+ learning map "🎯 Ready to start your learning journey?" 空态。
2. 在聊天框输入 "I want to learn Python" → 立刻看到 AI 回复 + 一道 Python 题 + 4 节点 Python 学习地图。
3. 刷新页面 → 状态保留（plan.goal / plan_concepts 都已经写回 DB）。
4. 切到另一个新账号 → 又是空态（resetUserState + plan 是空的）。

**经验（值得记下来）：**
- "覆盖式" fetch 模式（`if (data && data.length) { state.x = data }`）需要配合 "无条件先重置" 才稳。两条都要。
- 产品设计层面的"先用 demo 占位 vs 等用户输入"是常见权衡。本项目明确选了后者（chat-first），所以 onboarding 自动 attach 反而是反模式。
- 异步 persist 用 fire-and-forget + catch 兜底，是 UI 不阻塞 + DB 最终一致性的标准做法。
